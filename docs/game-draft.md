# 합성 게임 초안

## 1. 목표
- 웹브라우저 기반의 HTML + JavaScript 게임으로 구현한다.
- 전체 분위기는 동화책을 넘기는 듯한 아기자기한 색감과 배경을 사용한다.
- 핵심 루프는 `채집 → 연금 → 납품/판매 → 스토리/퀘스트 해금`으로 유지한다.
- 메인스토리와 퀘스트는 앞으로 계속 추가될 예정이므로, 처음부터 **데이터 추가만으로 확장 가능한 구조**를 전제로 설계한다.

## 2. 기본 화면 구성
메뉴는 다음 7개 탭으로 고정한다.
1. 메인스토리
2. 퀘스트
3. 채집
4. 연금
5. 판매
6. 컨테이너
7. 세이브

추가로 우측 또는 하단에 다음 공통 영역을 둔다.
- 현재 소지 콜
- 최근 행동 로그 10개
- 현재 진행 중인 메인스토리 / 퀘스트 요약
- 중요 해금 알림

## 3. 플레이 흐름 초안
### 초반 흐름
1. 플레이어가 게임을 시작한다.
2. 메인스토리 0100이 자동으로 열린다.
3. 메인스토리 0100 종료 후 퀘스트 0101 ~ 0113이 한꺼번에 등록된다.
4. 플레이어는 채집으로 재료를 모으고, 필요한 경우 연금으로 가공 아이템을 만든다.
5. 퀘스트를 완료해 보상을 받고 콜을 모은다.
6. 0101 ~ 0113 전체 완료 시 메인스토리 0200이 해금된다.
7. 메인스토리 0200 종료 후 퀘스트 0201이 등록된다.
8. 증류수 제작 및 납품을 통해 다음 스토리 분기로 넘어간다.

### 중장기 흐름
- 메인스토리는 `챕터 > 스토리 노드 > 대사 페이지` 구조로 확장한다.
- 퀘스트는 `스토리 해금형`, `재료 납품형`, `제작 확인형`, `특수 아이템 확보형` 등으로 확장 가능하게 둔다.
- 후반부에는 특정 아이템 제작, 특정 연금 계열 완성, 희귀 재료 확보 등 복합 조건 퀘스트도 추가할 수 있게 한다.

## 4. 메인스토리 설계 초안
### 표시 방식
- 비주얼 노벨처럼 한 번에 3~4줄만 표시한다.
- `다음` 버튼으로 다음 페이지를 넘긴다.
- 마지막 페이지를 읽으면 해당 메인스토리 노드가 `읽음` 상태가 된다.
- 읽기 완료 시 다음 메인스토리 또는 신규 퀘스트가 해금될 수 있다.

### 추천 데이터 구조
메인스토리는 지금처럼 문서로 기획하되, 실제 구현 시에는 아래 구조로 변환 가능한 형태를 추천한다.

```json
{
  "id": "0200",
  "title": "카페와 요정의 숲",
  "unlockConditions": [
    { "type": "questClearAll", "questIds": ["0101","0102","0103"] }
  ],
  "pages": [
    ["심부름을 모두 끝내고", "포포리와 주인공은", "새로 생긴 카페로 향한다."],
    ["카페 점장 에드몬드가", "직접 케이크를 서빙하고", "루치아의 부상을 이야기한다."]
  ],
  "rewards": {
    "unlockQuestIds": ["0201"]
  }
}
```

### 확장 원칙
- 스토리 ID는 `0100`, `0200`, `0300`처럼 챕터 단위로 증가시킨다.
- 한 스토리 안에는 여러 페이지를 넣되, 각 페이지는 3~4줄로 제한한다.
- 스토리 해금 조건은 단일 조건이 아니라 배열로 관리해, 나중에 복수 조건도 처리 가능하게 한다.
- 스토리 완료 보상은 퀘스트 해금, 아이템 지급, 콜 지급, 새 지역 오픈 등으로 확장 가능하게 한다.

## 5. 퀘스트 설계 초안
### 퀘스트 종류
초기 버전에서는 아래 4종을 우선 지원하면 확장성이 좋다.
1. **납품 퀘스트**: 재료 또는 제작 아이템을 제출
2. **제작 퀘스트**: 특정 아이템을 1회 이상 제작
3. **보유 퀘스트**: 특정 아이템을 가지고 있기만 하면 완료
4. **스토리 연계 퀘스트**: 특정 스토리를 읽어야 열리거나 완료됨

### 추천 데이터 구조
```json
{
  "id": "0201",
  "name": "먼 길을 위한 준비",
  "description": "요정의 숲에 가기 전에 증류수 1개를 준비하자.",
  "category": "delivery",
  "unlockConditions": [
    { "type": "storyRead", "storyId": "0200" }
  ],
  "objective": {
    "itemName": "증류수",
    "count": 1
  },
  "rewards": {
    "col": 100,
    "unlockStoryIds": ["0300"]
  }
}
```

### 확장 원칙
- 퀘스트 ID는 메인스토리와 연관이 보이도록 `0101`, `0102`, `0201`처럼 관리한다.
- 퀘스트는 `숨김`, `진행중`, `완료 가능`, `완료` 상태를 가진다.
- 앞으로 퀘스트 수가 늘어나도 관리하기 쉽도록 `chapter`, `region`, `npc`, `repeatable` 같은 보조 필드를 추가할 수 있게 둔다.
- 메인스토리 해금 조건으로 `특정 퀘스트 전부 완료`, `특정 카테고리 퀘스트 n개 완료`도 넣을 수 있게 한다.

## 6. 메뉴별 기능 초안
### 6-1. 메인스토리
- 현재 읽을 수 있는 스토리 목록만 노출한다.
- 아직 조건을 만족하지 못한 스토리는 숨긴다.
- 읽은 스토리는 다시보기 가능하게 한다.
- 최근 해금된 스토리는 `NEW` 표시를 준다.

### 6-2. 퀘스트
- 진행 중 / 완료 가능 / 완료됨 탭 또는 필터를 둔다.
- 각 퀘스트 카드에는 다음 정보를 표시한다.
  - 퀘스트 ID
  - 퀘스트 이름
  - 목표 아이템과 수량
  - 현재 보유 수량
  - 완료 보상
- 납품형 퀘스트는 버튼 클릭 시 필요한 수량만 차감하고 완료 처리한다.
- 이미 완료한 퀘스트는 별도 완료 목록에서 확인 가능하게 한다.

### 6-3. 채집
- 1회 채집마다 10콜을 소모한다.
- `idea/materials.json`의 확률표를 기준으로 1개 재료를 획득한다.
- 콜이 부족하면 채집 버튼을 비활성화한다.
- 로그에는 `채집 성공: 깨끗한 물 x1`처럼 남긴다.
- 장기적으로는 지역 선택 기능을 추가할 수 있도록 구조를 열어 둔다.

### 6-4. 연금
- 레시피 목록은 현재 해금된 것만 표시한다.
- 하위 재료를 한 번도 모두 모아본 적 없는 상위 레시피는 숨긴다.
- 한 번 해금한 레시피는 이후 재료가 없어도 목록에 남긴다.
- 클릭 1회당 아이템 1개를 제작하고, 재료를 즉시 차감한다.
- 제작 불가 시 부족한 재료를 표시한다.

#### 레시피 해금 규칙
- 레시피에 필요한 모든 재료 또는 중간 제작물을 **최소 1회 이상 보유**했을 때 해금 후보가 된다.
- 실제 제작 가능 여부와 별개로, 한 번 해금되면 영구 해금 상태를 유지한다.
- 이 규칙은 상위 연금 아이템이 많아져도 동일하게 적용한다.

### 6-5. 판매
- 현재 보유 중인 아이템만 판매 리스트에 노출한다.
- 재료와 제작 아이템을 함께 보여주되, 구분 섹션을 둔다.
- 판매는 1회 클릭당 1개씩 판매한다.
- 판매 가격은 `idea/items.json` 기준을 사용한다.
- 순수 재료의 판매가 필요하면 추후 `materials.json`에도 가격 필드를 추가할 수 있게 한다.

### 6-6. 컨테이너
- `재료` / `조합 아이템` 섹션을 나눠 표시한다.
- 이름, 수량, 최근 획득 여부를 보여준다.
- 수량 0인 항목은 기본적으로 숨긴다.
- 해금 도감 느낌을 원하면 `획득한 적 있음` 목록을 별도로 둘 수 있다.

### 6-7. 세이브
- 자동 저장을 기본으로 한다.
- 수동 저장 버튼을 별도로 둔다.
- 저장 초기화 버튼을 둔다.
- 초기화 시 확인 단계를 넣어 오작동을 막는다.
- 저장 데이터에는 최소 아래 값이 포함되어야 한다.
  - 콜
  - 인벤토리
  - 읽은 메인스토리
  - 열린 메인스토리
  - 진행 중/완료 퀘스트
  - 해금된 레시피
  - 최근 로그 10개

## 7. 상태 데이터 초안
게임 전체 상태는 아래처럼 관리하면 확장하기 쉽다.

```js
{
  col: 0,
  inventory: {
    materials: {},
    items: {}
  },
  stories: {
    unlocked: [],
    read: []
  },
  quests: {
    active: [],
    completable: [],
    completed: []
  },
  recipes: {
    unlocked: []
  },
  logs: []
}
```

### 상태 관리 원칙
- `unlocked`와 `read`, `active`와 `completed`를 분리해 관리한다.
- 스토리/퀘스트가 많아질수록 탐색 비용이 커지므로, ID 기반 맵 구조도 고려한다.
- 저장은 `localStorage`를 우선 사용하고, 추후 export/import 기능을 덧붙일 수 있게 한다.

## 8. 로그 설계 초안
- 최신 10개만 유지한다.
- 새 로그가 추가되면 가장 오래된 로그를 제거한다.
- 로그 종류 예시
  - 채집 결과
  - 제작 성공/실패
  - 퀘스트 완료
  - 메인스토리 해금
  - 저장 완료
- 로그 문구는 짧고 누가 봐도 바로 이해되게 고정 템플릿을 사용한다.

## 9. 향후 추가를 고려한 데이터 운영 방안
### 메인스토리 추가 시
- `idea/mainstory.md`는 기획 원본으로 유지한다.
- 실제 게임에서는 구조화된 JSON 또는 JS 데이터로 파싱해 사용하는 방식을 권장한다.
- 새 메인스토리 추가 시 필요한 것은 아래 3가지로 통일한다.
  1. 스토리 ID
  2. 해금 조건
  3. 완료 시 열릴 퀘스트 또는 다음 스토리

### 퀘스트 추가 시
- `idea/quests.md`도 장기적으로는 구조화 데이터로 옮기는 것이 좋다.
- 새 퀘스트는 `해금 조건`, `목표`, `보상`만 채우면 동작하도록 설계한다.
- 이렇게 하면 퀘스트가 수십~수백 개로 늘어나도 UI와 로직을 거의 바꾸지 않아도 된다.

### 추천 확장 포인트
- 지역별 채집 테이블 분리
- NPC 호감도 연동 퀘스트
- 챕터별 배경 일러스트 변경
- 특정 아이템 제작 시 전용 스토리 컷신 재생
- 반복 퀘스트 / 일일 퀘스트
- 엔딩 분기 조건

## 10. 초기 구현 우선순위
### 1차
- 기본 레이아웃과 7개 메뉴 탭
- 자동 저장 / 수동 저장 / 초기화
- 채집 기능
- 컨테이너 표시
- 로그 10개 제한

### 2차
- 연금 레시피 표시 및 제작
- 레시피 해금 처리
- 판매 기능
- 퀘스트 납품 처리

### 3차
- 메인스토리 페이지 연출
- 스토리 완료에 따른 퀘스트 해금
- 챕터 진행 흐름 연결

### 4차
- 데이터 구조 정리
- 향후 스토리/퀘스트 대량 추가를 위한 JSON 분리
- UI 꾸미기 및 동화책풍 연출 강화

## 11. 한 줄 요약
이 초안의 핵심은 **현재 주어진 메인스토리/퀘스트를 바로 구현할 수 있으면서도, 이후 스토리와 퀘스트를 데이터만 추가해 확장할 수 있는 구조를 먼저 잡는 것**이다.
